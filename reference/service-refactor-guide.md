# 服务化重构参考：从脚本到可交付服务（框架无关）

> 供 `plan-breakdown` 在遇到「脚本/模块要改造成对外交付的服务」类需求时按需引用。
> 提炼自架构通识（参考案例：AI 应用开发课程第 05 课），所有原则跨语言/跨框架成立。

## 核心问题

「模块能跑通」≠「别人能用上」。差的不是 AI 实现质量，是**交付能力**：
接口契约（承诺了什么）、封装边界（内部不外露）、职责分层（代码怎么摆）。

## 三个概念（判据形式）

1. **接口契约**：输入是什么 / 输出与错误是什么 / 怎么调用。三问答清即为契约；载体用行业标准（OpenAPI、类型化 Schema）。
2. **封装边界**：调用方代码里出现内部实现词（框架名/模型名）= 边界泄漏。
3. **职责分层**（每层「不该做什么」）：
   - 接入层不该：调模型、写 prompt、做业务判断
   - 业务层不该：感知协议（HTTP 等）、直接读环境变量
   - 配置层不该：混业务逻辑
   - 判据：**一个小改动要同时改三处 = 分层错了**

## 渐进演进路线（按关注点逐个剥离）

每步只剥离一个关注点，**每步结束代码可运行、可验证**。顺序和步数按项目实际取舍：

| 阶段 | 剥离什么 | 动作 | 完成判据 |
|---|---|---|---|
| v0 → v1 | 模型/环境配置 | 写死的 endpoint/key → 环境变量 + `.env`（不进 git）+ `.env.example`（进 git） | 只改 `.env` 就能切换模型/环境，代码零改动 |
| v1 → v2 | prompt/资源 | 代码里的 prompt 字面量 → 外部模板文件（YAML/模板引擎） | 改 prompt 不需要动源代码；prompt 可 git diff、可 A/B |
| v2 → v3 | 装配边界 | 散落的顶层逻辑 → 工厂函数 + 包结构；稳定依赖启动时注入一次，请求级数据每次调用传入 | 流程可被任何入口（CLI/HTTP）复用；单元可独立测试 |
| v3 → v4 | 接入协议 | 加 HTTP（或 gRPC 等）接入层：路由 + 类型化 Schema | 接入层只做「translate 协议 ↔ 内部调用」三件事；业务文件零改动 |

配置三层覆盖（v1 的完整形态）：

```
代码默认值（兜底） ← 配置文件（本地开发，不进 git） ← 环境变量（部署注入，最高优先级）
```

## 可消费性检查（v4 后的「做好了」标准）

- 健康检查端点：证明服务存活，**不依赖任何业务模块**
- 自动文档：由类型化契约自动生成（如 OpenAPI /docs）
- 错误码语义：4xx = 调用方的错，5xx = 服务方的错
- 版本前缀：`/v1/...`——契约一经发布不改，要变加 v2

## 语言对位表（架构动作不变，只换工具）

| 动作 | Python | Go | Node.js | Java |
|---|---|---|---|---|
| 配置加载 | dotenv + os.environ | viper | dotenv + process.env | Spring profiles |
| 契约定义 | Pydantic | struct tag + validator | zod / TypeBox | Bean Validation |
| HTTP 接入 | FastAPI | Gin / Echo | Express / Fastify | Spring Boot |
| 模板机制 | jinja2 / YAML | text/template | handlebars | FreeMarker |

## 反模式

- 一把梭重构（同时改配置+prompt+结构+协议）→ 出错无法定位
- 接入层里写业务判断/调模型 → 换协议时业务跟着重写
- 健康检查里调业务/数据库 → 不再是「服务自身存活」的证据
- 真实 key 进 git → 重写历史 + 吊销 key，代价极高
